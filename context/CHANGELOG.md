## notifications@ sender + lock down from_email (completed 2026-07-21)
Two related mail-safety fixes, split into two concerns/commits per the original plan.

**Concern 1 — split sender config from reply config.** Outbound RYZE mail was going out as `dane@ryze.ai`; the goal was a neutral infrastructure address without moving reply-to/support routing away from the real inbox. Added `REPLY_TO_EMAIL` setting (`app/core/config.py`), defaulting to empty so a missing env var reproduces the old behavior exactly (`branding.py`'s `_ryze_defaults` falls back to `FROM_EMAIL` when unset). Server env set to `FROM_EMAIL=notifications@ryze.ai`, `REPLY_TO_EMAIL=dane@ryze.ai`, `ADMIN_EMAIL=dane@ryze.ai`.

**Concern 2 — remove from_email from tenant settings.** Tenant admins could PATCH `from_email` on their own branding via `/api/settings/tenant` to any address; since Resend only delivers from verified domains and only `ryze.ai` is verified, an unverified value there silently broke **all** of that tenant's outbound mail — this had already happened live with Green Path. Fixed by having the PATCH handler explicitly skip `from_email` (with a `logger.info` per skip) instead of persisting it, and removing it from the response model. Field stays on the request model (`extra="forbid"` unchanged) so the still-deployed frontend doesn't 422; the skip is removable once the frontend stops sending it. No migration — `tenant.from_email` column and `get_branding`'s read of it stay in place for a future verified-domain model.

## Match score calibration & distance-operator unification (completed 2026-07-21)
Match *ranking* was correct everywhere, but displayed scores were misleadingly low on the candidate side (a strong match showed 12%). Root cause: candidate-side paths computed `score = 1 - L2_distance` (`<->`), and L2 between unit-norm OpenAI embeddings runs ~0.85-1.0 even for strong matches, crushing all scores toward 0. The employer-side path used cosine (`<=>`), so the two sides also disagreed on the same pairing. This unified the operator and calibrated the displayed score — the deferred refactor logged in earlier History (shared helper + `<->`/`<=>` unification).

Created one shared scoring helper (`app/services/matching.py`, `SIM_FLOOR`/`SIM_CEIL` constants + `compute_match_score(cos_distance)`) used by all three match paths: `GET /api/candidates/me/job-matches`, `GET /api/job-orders/{id}/candidate-matches`, and `tool_match_jobs_to_candidate` in chat.py. All three now use cosine distance (`<=>`) in their raw SQL — `job_orders.py` already did, `candidates.py` and `chat.py` were switched from `<->`. Score formula: `cos_sim = 1.0 - cos_distance`, then `clamp((cos_sim - 0.25) / (0.75 - 0.25), 0.0, 1.0)`, rounded to 4 decimals. The transform is monotonic so ranking is provably unchanged; response field name (`match_score`) and schema are unchanged. Removed the dead `round(max(0.0, 1.0 - distance), 4)` pattern from all three call sites.

Deliberately left untouched: `tool_search_candidates`/`tool_search_employers`/`tool_search_job_orders`/`_vector_search` in chat.py still use the old uncalibrated `1.0 - distance` formula over `<=>` — query-text-vs-profile similarity has a different natural range than profile-vs-job similarity and would need its own floor/ceiling constants.
