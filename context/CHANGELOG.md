## Job Order PDF â€” banner render parity with Employer Profile PDF + tenant-branded footer (completed 2026-07-24)
Comparing a Greenscene Landscaping job order PDF against its employer profile PDF (both generated 2026-07-23) surfaced two defects. The footer was hardcoded to `RYZE.AI` / "Job Order Brief" as literals in `PDF_HTML` â€” the last hardcoded-identity holdout after the candidate and employer PDFs were already migrated to the `get_branding` resolver. The banner was visibly over-cropped relative to the employer PDF, traced to `.banner` rendering at `height: 180px` against the employer template's `.banner-strip` at `140px`, plus divergent overlap offsets (`-44px` vs `-42px`) and gradient-fallback stop counts (3-stop vs 2-stop).

Footer fixed the same way as the candidate/employer PDFs: `{footer_brand}` placeholder fed by `get_branding(db, tenant_id)` in the PDF route, tagline removed entirely (not parameterized) to match their already-shipped end state. Banner shipped as full parity with the employer template â€” `height: 140px`, `-42px` identity-zone overlap, 2-stop gradient fallback â€” byte-identical to `.banner-strip` in every CSS property that affects the rendered crop.

**The banner height wasn't a one-shot fix â€” production testing against the real Greenscene image produced a genuine evidence trail.** 140px crops the bottom of the banner photo strip; the original 180px and a subsequently-tried 220px both cropped differently â€” the banner's script/logo content along the top and right edge instead, and more aggressively than 140px, since a taller `cover`-fit box requires a larger scale factor and therefore *more* total crop, not less. None of the three heights render the full composition (both the full script and the full photo strip) simultaneously. 140px was the value ultimately kept, not because it's crop-free, but because it reproduces the exact trade-off already live and approved on the employer PDF for the same image â€” genuine parity, not a coincidence of matching numbers. **This is a structural limitation of fixed-height + `background-size: cover`, not a bug in this fix** â€” a real fix needs width-driven intrinsic-aspect rendering, a stored focal point, or an enforced banner aspect ratio at upload, and is unscoped follow-up work, not attempted here.

Also flagged, not fixed (per explicit out-of-scope instruction): three functionally-equivalent but differently-sourced tenant-derivation patterns coexist across the codebase â€” `job_orders.py`'s literal `"ryze"`, `employer_profiles.py`'s imported `RYZE_TENANT` constant, and `candidates.py`'s `_tenant(user)` helper. All resolve to the same value today; normalizing them is parked for a future session.

## Job Orders â€” hourly rate range + employment type + industry-agnostic parser (completed 2026-07-23)
The job order model, parser, and export pipeline all assumed a salaried, direct-hire role â€” no way to represent an hourly contract or temp-to-hire engagement, and the AI parser's prompt was implicitly finance-flavored. This extended job orders to support any industry and any engagement type: hourly rate range alongside the existing annual salary, an `employment_type` field (`contract` | `contract_to_hire` | `direct_hire`), an industry-agnostic parser that extracts both, and the new fields flowing through to embedding text (so RYZE Intelligence can match on engagement type) and the PDF export (chips + sidebar). A companion frontend task ships after this repo, per the established Pydantic-tolerates-missing-fields-either-direction deploy order.

Shipped as 4 commits, one concern each. **Commit 1** (`5edb836`) â€” `hourly_min`/`hourly_max` as `Numeric(8,2)` (chosen over `Integer` because hourly rates commonly carry cents) and `employment_type` as `String(50)` on `JobOrder`; all three fields added to all five classes in `app/schemas/job_order.py`; migration `039ab00599f3`, nullable/no-backfill, verified with a full local upgrade â†’ `\d job_orders` â†’ `downgrade -1` â†’ re-upgrade round trip. **Commit 2** (`daf5cbe`) â€” `parse_job_description`'s prompt rewritten to open "from any industry" and extract the two new fields with explicit phrasing-mapping rules (temp/1099 â†’ `contract`, temp-to-hire â†’ `contract_to_hire`, permanent/full-time â†’ `direct_hire`) and a hard rule against ever converting between hourly and annual figures. **Commit 3** (`388101f`) â€” `build_job_order_text` appends hourly range + a human-readable employment-type label to the embedding text, correctly handling all three hourly cases (min-only, max-only, both) unlike the pre-existing salary block, which only handles the min-only case (left untouched, out of scope); `fmt_hourly` + `EMPLOYMENT_TYPE_LABELS` added to the job order PDF template and wired into both the PDF chip row and the sidebar Job Details card. **Commit 4** (`264fa01`) â€” both `JobMatchResult` constructions in `candidates.py` (ranked + unranked fallback) now surface the new fields to candidates, not just admins/employers.

`audit_tenant_coverage.py` showed no new REVIEW/HARDCODED lines throughout (same 2 pre-existing unrelated `candidates.py` findings). Deployed to production (https://api.ryze.ai): live `GET /openapi.json` confirmed the new fields on `JobOrderResponse`/`JobOrderCreate`/`JobMatchResult`, and the user confirmed the migration via `psql \d job_orders` directly against the production DB. ## DB Explorer â€” platform-owner global visibility (completed 2026-07-22)
The DB Explorer is the superadmin's window into the entire database, but every endpoint tenant-filtered on `_tenant(current_user)` â€” the superadmin's own resolved tenant â€” with only one ad-hoc bypass, for the `users` table. Live symptom: the database held 6 `job_orders` and 2 `bookings`, all belonging to `green_path_recruiting`, but the explorer showed zero for both to `dane@ryze.ai` (tenant `ryze`). The intended fix already existed as dead code: a `_is_platform_owner()` helper with a docstring describing exactly the desired behavior, but never called by any endpoint.

**Goals:** (1) platform owner (`is_superuser` AND `tenant_id == 'ryze'`) sees everything, unscoped, across every DB Explorer endpoint â€” counts, browse, export, PATCH, DELETE; (2) everyone else (a hypothetical firm-level superuser) stays strictly tenant-scoped, with no behavior change anywhere else in the app; (3) tenant attribution stays visible â€” `tenant_id` surfaced in frontend summary columns so the platform owner can see whose row is whose.

Wired `_is_platform_owner()` into all five `app/api/db_explorer.py` endpoints, replacing the `users`-only special case with the general rule (`table in TENANT_SCOPED_TABLES and not unscoped`). This is a deliberate behavior tightening for a hypothetical firm-level superuser: they now see only their own tenant's `users` rows instead of every tenant's, which the old bypass allowed. Discovered along the way that `bookings` was in `TENANT_SCOPED_TABLES` but missing `tenant_id` from `TABLE_COLS` â€” confirmed the column exists on the model and added it, rather than removing `bookings` from the scoped set. Frontend companion change surfaced `tenant_id` in `SUMMARY_COLS` for the five affected tables. Zero changes outside the two files; `get_current_tenant`/`get_current_admin_tenant`/`_check_tenant_access` untouched.

## Match score calibration & distance-operator unification (completed 2026-07-21)
Match *ranking* was correct everywhere, but displayed scores were misleadingly low on the candidate side (a strong match showed 12%). Root cause: candidate-side paths computed `score = 1 - L2_distance` (`<->`), and L2 between unit-norm OpenAI embeddings runs ~0.85-1.0 even for strong matches, crushing all scores toward 0. The employer-side path used cosine (`<=>`), so the two sides also disagreed on the same pairing. This unified the operator and calibrated the displayed score â€” the deferred refactor logged in earlier History (shared helper + `<->`/`<=>` unification).

Created one shared scoring helper (`app/services/matching.py`, `SIM_FLOOR`/`SIM_CEIL` constants + `compute_match_score(cos_distance)`) used by all three match paths: `GET /api/candidates/me/job-matches`, `GET /api/job-orders/{id}/candidate-matches`, and `tool_match_jobs_to_candidate` in chat.py. All three now use cosine distance (`<=>`) in their raw SQL â€” `job_orders.py` already did, `candidates.py` and `chat.py` were switched from `<->`. Score formula: `cos_sim = 1.0 - cos_distance`, then `clamp((cos_sim - 0.25) / (0.75 - 0.25), 0.0, 1.0)`, rounded to 4 decimals. The transform is monotonic so ranking is provably unchanged; response field name (`match_score`) and schema are unchanged. Removed the dead `round(max(0.0, 1.0 - distance), 4)` pattern from all three call sites.

Deliberately left untouched: `tool_search_candidates`/`tool_search_employers`/`tool_search_job_orders`/`_vector_search` in chat.py still use the old uncalibrated `1.0 - distance` formula over `<=>` â€” query-text-vs-profile similarity has a different natural range than profile-vs-job similarity and would need its own floor/ceiling constants.
